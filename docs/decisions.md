## 2026-01-16 — 현재 위치 탐지 안정화(Geolocation 재시도 전략)

- **결정**:
  - `getCurrentPosition`을 1번만 호출하지 않고, 옵션을 바꿔가며 **최대 3번 재시도**하도록 구현
    - 캐시 기반 빠른 시도 → 기본 시도 → 고정밀(high accuracy) + 긴 타임아웃 시도
- **이유**:
  - 권한이 허용(granted)이어도 로컬 환경/네트워크 상태에 따라 `TIMEOUT`이 자주 발생할 수 있기 때문에, 사용자 경험을 안정화할 필요가 있음
- **비용**:
  - 구현 복잡도 증가(옵션/재시도 흐름)
  - 일부 환경에서 요청 시간이 길어질 수 있음(마지막 고정밀 시도)

## 2026-01-16 — 현재 위치명(reverse geocoding) api 교체

- **결정**:
  - Open‑Meteo geocoding reverse 엔드포인트(404) 대신 **BigDataCloud reverse geocoding** API로 위치명(label) 조회
- **이유**:
  - 브라우저에서 바로 호출 가능한(키 없이) reverse geocoding이 필요했고, 기존 URL이 404로 실패했음
  - 현재 위치 카드에 “좌표” 대신 사람이 읽을 수 있는 “지명”을 표시해 UX를 개선하기 위함
- **비용**:
  - 외부 의존성 증가(추가 API 공급자)
  - 응답 스키마/정책이 바뀌면 추후 교체 비용이 발생할 수 있음

## 2026-01-16 — 현재 위치 fallback 전략(Geolocation → IP → 마지막 위치 → 기본값)

- **결정**:
  - Geolocation이 거부/실패될 경우, **IP 기반 대략 위치**로 한 번 더 시도
  - IP도 실패하면 **마지막 저장 위치(localStorage)** 를 사용함
- **이유**:
  - 위치 권한을 거부하거나 환경 문제로 위치 탐지가 실패해도 앱이 “빈 화면”이 되지 않도록 하기 위함
  - 초기 진입 UX를 안정화하고, 날씨/검색/즐겨찾기 흐름을 끊지 않기 위함
- **비용**:
  - IP 기반 위치는 정확도가 낮아 오차가 발생할 수 있음(“대략 위치” 표기 필요)
  - 외부 의존성 증가(IP geolocation API)
