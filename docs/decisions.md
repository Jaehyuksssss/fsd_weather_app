- **결정**:
  - 레이어 기반으로 `app/pages/widgets/features/entities/shared` 로 폴더 구조를 나눴음
- **이유**:
  - 규모가 커질수록 “무엇이 어디에 있어야 하는지”를 팀 단위로 빠르게 합의하기 위해(가정)
  - UI/라우팅/Provider/도메인 로직의 책임을 분리해 변경 영향 범위를 줄이기 위함
  - 과제 요구사항(기능 확장: 검색/즐겨찾기/상세 페이지 등)에 맞춰 점진적으로 기능을 추가하기 위함
- **비용**:
  - 오히려 복잡도가 증가 할 수 있음

## 2026-01-16 — 현재 위치 탐지 안정화(Geolocation 재시도 전략)

- **결정**:
  - `getCurrentPosition`을 1번만 호출하지 않고, 옵션을 바꿔가며 **최대 3번 재시도**하도록 구현
    - 캐시 기반 빠른 시도 → 기본 시도 → ip 기반 + 긴 타임아웃 시도
- **이유**:
  - 권한이 허용(granted)이어도 로컬 환경/네트워크 상태에 따라 `TIMEOUT`이 자주 발생할 수 있기 때문에, 사용자 경험을 안정화할 필요가 있음
- **비용**:
  - 구현 복잡도 증가(옵션/재시도 흐름)
  - 일부 환경에서 요청 시간이 길어질 수 있음(마지막 고정밀 시도)

## 2026-01-16 — 현재 위치명(reverse geocoding) api 교체

- **결정**:
  - Open‑Meteo geocoding reverse 엔드포인트(404) 대신 **BigDataCloud reverse geocoding** API로 위치명(label) 조회
- **이유**:
  - 브라우저에서 바로 호출 가능한(키 없이) reverse geocoding이 필요했고, 기존 URL이 404로 실패했음
  - 현재 위치 카드에 “좌표” 대신 사람이 읽을 수 있는 “지명”을 표시해 UX를 개선하기 위함
- **비용**:
  - 외부 의존성 증가(추가 API 공급자)
  - 응답 스키마/정책이 바뀌면 추후 교체 비용이 발생할 수 있음

---

## 2026-01-16 — 현재 위치 fallback 전략(Geolocation → IP → 마지막 위치 → 기본값)

- **결정**:
  - Geolocation이 거부/실패될 경우, **IP 기반 대략 위치**로 한 번 더 시도
  - IP도 실패하면 **마지막 저장 위치(localStorage)** 를 사용함
- **이유**:
  - 위치 권한을 거부하거나 환경 문제로 위치 탐지가 실패해도 앱이 “빈 화면”이 되지 않도록 하기 위함
  - 초기 진입 UX를 안정화하고, 날씨/검색/즐겨찾기 흐름을 끊지 않기 위함
- **비용**:
  - IP 기반 위치는 정확도가 낮아 오차가 발생할 수 있음(“대략 위치” 표기 필요)
  - 외부 의존성 증가(IP geolocation API)

---

## 2026-01-16 — 검색 라벨 → 좌표 지오코딩 전략(Open‑Meteo → Nominatim fallback + 캐싱)

- **결정**:
  - 검색 선택된 `label`을 좌표로 변환할 때, 1차로 **Open‑Meteo geocoding(search)** 를 사용
  - 결과가 없으면 2차로 **Nominatim(OSS) geocoding** 을 fallback으로 사용
  - 성공한 경우만 `localStorage`에 `label → coords`를 캐싱해서 반복 호출을 줄임
- **이유**:
  - `korea_districts.json`의 행정구역/법정동 문자열은 Open‑Meteo 데이터에서 누락되는 경우가 있어 “없음” 비율을 낮출 필요가 있음
  - 과제 범위에서 빠르게 커버리지를 확보하고, 사용자 경험을 안정화하기 위함
- **비용**:
  - 외부 의존성 증가(추가 지오코딩 공급자)
  - Nominatim은 사용 정책/요청량 제한이 있어 남용 방지(캐싱/선택 시에만 호출)가 필요함

## 2026-01-16 — 즐겨찾기 “순서 편집 모드” 도입(일반 모드/편집 모드 분리)

- **결정**:
  - 즐겨찾기 목록에서 드래그&드롭 정렬을 항상 노출하지 않고, “순서 편집” 버튼으로 **편집 모드에서만** 활성화
  - 편집 모드에서는 상세 이동/별칭/삭제 등 다른 상호작용을 숨기거나 제한하고, **정렬 작업에만 집중**하도록 구성
- **이유**:
  - 드래그 핸들은 발견성이 낮고(숨음), 일반 클릭(상세 이동)과 충돌이 잦아 UX가 불안정해질 수 있음
  - “정렬하고 싶다”는 사용자 의도를 명확히 분리해 실수를 줄이기 위함
- **비용**:
  - 모드 상태에 따른 UI/이벤트 분기가 추가되어 구현 복잡도가 증가함

## 2026-01-16 — 즐겨찾기 정렬 DnD를 Pointer Events + Pointer Capture로 구현

- **결정**:
  - 외부 DnD 라이브러리 없이 pointer 기반으로 정렬을 구현하고, 드래그 시작 요소에 `setPointerCapture`를 사용
- **이유**:
  - 모바일/트랙패드 환경에서 드래그 중 포인터가 영역 밖으로 벗어나도 `pointermove/up`를 안정적으로 수신하기 위함
  - “순서 편집” 모드에서는 카드 전체를 드래그 대상으로 삼아 조작성을 높이기 위함
- **비용**:
  - 키보드/스크린리더 친화적인 재정렬 UX는 추가 설계가 필요함(향후 개선 과제)

## 2026-01-16 — 시간대별 스크롤: 드래그 스와이프 + 화살표 버튼 병행

- **결정**:
  - 시간대별 가로 리스트를 `SwipeScroll`로 공통화하고,
    - 터치/마우스 드래그로 스와이프
    - 좌/우 화살표 버튼으로도 이동
  - 화살표는 “시간대별” 헤더 라인과 같은 층에서 좌/우에 배치하고, 시작/끝에서는 자동 disabled 처리
- **이유**:
  - 모바일(스와이프)과 데스크탑(버튼 클릭)에서 모두 조작성 편의성을 확보하기 위함
  - 스크롤바 의존을 줄이고, 카드 레이아웃 내에서 명확한 내비게이션을 제공하기 위함
- **비용**:
  - 스크롤 경계 판정(atStart/atEnd)이 서브픽셀/스냅 등으로 흔들릴 수 있어 보정(EPS)이 필요함
  - UI 요소가 늘어 레이아웃/일관성 관리 비용이 증가함

## 2026-01-16 — 배포

- **결정**:
  - 정적 배포를 위해 s3 와 AWS Cloudfront를 사용
- **이유**:
  - Vite 빌드 산출물(`dist/`)은 정적 파일이므로, 객체 스토리지 + CDN 조합이 단순하고 안정적임
  - CloudFront 캐시로 응답 속도를 확보하고, 트래픽이 늘어도 수평 확장이 쉬움
  - 운영 비용을 예측 가능하게 가져가고(요청/전송량 기반), AWS 표준 구성으로 관리하기 위함
- **비용**:
  - 프론트만으로 해결하기 어려운 “서버 기능”이 필요해지면 별도 백엔드/서버리스 구성이 추가로 필요함
